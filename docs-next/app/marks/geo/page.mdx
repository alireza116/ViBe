# Geo

<Lead>Geographic marks for map elicitation. Set chart-level `projection` (Plot-style: `"mercator"`, `{ type, domain, … }`, or a live d3.geo* instance). Position uses **apply / invert** on that shared projection — not x/y scales. Provide basemap topology with `geoBasemap({ geojson })` (a GeoJSON FeatureCollection / Feature / Geometry). Editable overlays live on the **dataset**. Examples below use Vancouver’s 22 local areas (`vancouver` in the docs scope — replace with your own file).</Lead>

<Section id="tiles" title="Real basemaps (OSM raster tiles)">

For an actual map under your data — streets, water, labels — use `geoTile()`: the Leaflet model, a pyramid of 256px images from a tile server, laid on the plane behind the marks. **It requires `projection: "mercator"`** (unrotated). A tile is a pre-rendered picture baked in Web Mercator, so under any other projection it cannot be made to register with the data; the mark checks the projection's behaviour, draws nothing, and warns rather than showing you a misaligned map. Everything else is unchanged — the same lon/lat marks and the same edit.geo.* gestures ride on top, because they still go through the same projection. **Attribution is a licence condition** of every tile service and is drawn by default. Public OSM tiles are rate-limited and not meant for production traffic: point `url` at your own server or a paid provider when you ship.

<Section id="labels" title="Label and rename places">

A label on a map is an ordinary **text** mark that happens to be positioned by the projection — so `geoText` is the only geo-specific part, and every text behaviour comes along free: `editText()` to retype the content, `cycle()` to advance a categorical label, `rotate()` on `angle`. Give it its own feature next to a geoPoint and the two read the same rows: drag the dot and the name follows. `dx`/`dy` nudge the glyph off the anchor without touching the lon/lat a drag writes.

<Section id="connected" title="Connected scatter over the map">

Give geoLine `lon`/`lat` channels instead of `coordinates` and it connects the dataset's **rows** into one path — the geographic sibling of the `path` mark. Points ride on a separate geoPoint feature, so dragging a stop reshapes the trail: the line re-derives from the same rows on the next render. Order defaults to `"sequence"` (array order); pass a field name to sort by it, or `series` to draw one trail per group.

<Example
  code={aRouteAcrossVancouver}
  title="A route across Vancouver"
  blurb="Five stops connected in `stop` order. The dots are a geoPoint mark with edit.geo.drag() — the trail follows because both marks read the same dataset."
  try={(
    <>
      <b>Drag</b> a stop to reroute; <b>click</b> the map to append one.
    </>
  )}
/>

</Section>

<Section id="rects" title="Geographic rectangles">

Boxes are axis-aligned in lon/lat (they may look slightly curved under Mercator). Grab an **edge** or **corner** to reshape, the **body** to move, and click empty space to mint another. The zone you grabbed is latched at dragstart by the `geoBrush` driver and held for the whole gesture, so a move never turns into a resize halfway through; the cursor tells you which zone you're in before you press.
