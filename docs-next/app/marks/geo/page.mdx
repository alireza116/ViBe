# Geo

<Lead>Geographic marks for map elicitation. Set a chart-level `projection` — `"mercator"`, `{ type, domain, … }`, or a live `d3.geo*` instance. Position goes through that projection's apply/invert, not x/y scales. Provide basemap topology with `geoBasemap({ geojson })`, a GeoJSON FeatureCollection, Feature, or Geometry. Editable overlays live on the dataset, like every other mark. Examples below use Vancouver's 22 local areas (`vancouver` in the docs scope) — swap in your own file.</Lead>

<ApiReference entries={api} />

<Section id="basemap" title="Providing basemap data">

The basemap is chrome. Pass GeoJSON into `geoBasemap`, and fit the projection with that same object as `domain` so the map fills the frame. Below, `vancouver` is the docs sample (22 local areas) — your app would import or fetch its own.

<Section id="tiles" title="Real basemaps (OSM raster tiles)">

`geoTile()` lays a raster basemap — a pyramid of 256px tiles from a tile server — behind your marks. Tiles are pre-rendered in Web Mercator, so it requires `projection: "mercator"`; other projections draw nothing and log a warning instead of a misaligned map. Your lon/lat marks and `edit.geo.*` gestures work the same on top, since they still go through the same projection. Attribution is a licence condition of every tile service, and is drawn by default. Public OSM tiles are rate-limited — point `url` at your own server or a paid provider in production.

<Section id="labels" title="Label and rename places">

`geoText` is a text mark positioned by the projection instead of x/y scales — that's the only geo-specific part, so every other text behaviour carries over: `editText()` to retype the content, `cycle()` to advance a categorical label, `rotate()` on `angle`. Pair it with a `geoPoint` as a separate mark reading the same rows, and dragging the dot moves the label along with it. `dx`/`dy` offset the glyph from its anchor without touching the lon/lat a drag writes.

<Section id="connected" title="Connected scatter over the map">

Give `geoLine` `lon`/`lat` channels instead of `coordinates` and it connects the dataset's rows into one path, the geographic sibling of `path`. Points are a separate `geoPoint` mark reading the same rows, so dragging a stop re-derives the trail on the next render. Order defaults to `"sequence"` (array order); pass a field name to sort by it, or `series` to draw one trail per group.

<Example
  code={aRouteAcrossVancouver}
  title="A route across Vancouver"
  blurb="Five stops connected in `stop` order. The dots are a geoPoint mark with edit.geo.move() — the trail follows because both marks read the same dataset."
  try={(
    <>
      <b>Drag</b> a stop to reroute; <b>click</b> the map to append one.
    </>
  )}
/>

</Section>

<Section id="rects" title="Geographic rectangles">

Boxes are axis-aligned in lon/lat — they can look slightly curved under Mercator. Grab an edge or corner to reshape, the body to move, or click empty space to mint another. The zone you grab is fixed for the whole gesture, so a move never turns into a resize halfway through. The cursor shows which zone you're in before you press.
