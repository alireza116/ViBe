# External controls

<Lead>Bind an edit to any external controller — a `<select>`, sliders, a rotate icon, a plain function — and drive it from outside the SVG. Give the edit a `name`, then call `el.control(name).set(value)`. Nothing about how edits work changes: the control synthesizes the **same** events a pointer produces and hands them to the one dispatch, so constraints, undo/redo, guides, and `on("change")` all run identically. It is a new input *source*, not a second interaction system — the same move the keyboard already makes when it turns an arrow key into a drag.</Lead>

<ApiReference entries={api} />

<Section id="how" title="Name an edit, then drive it">

The value flows the opposite way through the same scale. An edit maps a gesture → data by *inverting* a channel's scale; `control(name).set(value)` maps data → a gesture by running that scale *forward*, then dispatches the ordinary edit. For a value edit (`set`/`editText`) there is no pixel at all — the value is carried straight through. `control(name).accepts()` reports what the channel's scale allows, so a picker or slider can offer only valid values.

</Section>

<Section id="scatter" title="A picker and a drag pad">

The `<select>` writes the point's category through a `set()` edit; its options come from `control("category", 0).accepts().values`. The pad drives a two-axis `drag` live — `begin() … set({x,y}) … end()` collapses into one undo entry. The `clamp` on `y` gates the pad exactly as it gates a mouse drag: external control is not a bypass.

<Section id="pick" title="Pick vs. cycle: driving other gestures">

`.set(value)` covers the value edits (`set`/`editText`) and the positional ones (`drag`/`rotate`/`resize`). The click-triggered edits — `cycle`, `remove`, `toggle` — take no value; they just fire. `.fire()` drives those, dispatching the edit's own gesture on the datum. So one circle can offer a **picker** (`set` — jump straight to a value) *and* a **cycle** (`fire` — step to the next), on the same field, alongside the native click.

<Section id="face" title="Sliders for facial features">

Each facial parameter is a channel; bind it to a field with a named `set()` edit and a slider can write it — while the face stays directly draggable (grab the mouth, pull a smile). One dataset, two input sources, the same edits. Each slider's range comes from `control(field).accepts().domain`.

<Section id="rotate" title="A rotate icon for a box">

`rotate` reads a pointer angle about the mark; driving it by degrees is the inverse. `control("spin", 0).set(deg)` forward-encodes the angle to the pointer that would produce it, then runs the ordinary rotate edit. The box also rotates by direct drag.

<Section id="selection" title="Driving the selection">

Selection is **pipeline state**, not a data column — so it has its own external API beside `getData`/`setData`, and its own `select` event. `el.select(index)` picks a **specific item**; `el.selectWhere(field, value)` picks by **category** (the first row matching). Both go through the same path a click on a mark does (`edit.select()`), so the selected row gets the highlight outline and a legend picker edits it — whichever input moved the selection. `on("select")` reports the change; `getData` and `on("change")` stay silent, because no belief data moved.
