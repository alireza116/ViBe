# Authoring SDK

<Lead>Build new elicitation devices without forking the engine. Marks use `plot.encodeChannel` / `resolveStyle`; edits use `edit.makeEdit` + `invertChannel`; multi-event lifecycles register via `edit.registerDriver`. Attach edits on a channel (`y: { field, edit: move() }`) or at mark level (`edits: [...]`) — never as a fake `channels.edit` key. For the model these extend, see [Core concepts](/concepts); for the stock catalogue, see [Editing](/editing).</Lead>

<ApiReference entries={api} />

<Section id="custom-mark" title="A custom mark">

A mark is a view over the chart’s one dataset. The factory returns a feature object; the only required method is `build(data, scales, width, height)`. Call `normalizeMarkOptions` first, resolve every channel through `encodeChannel` / `resolveStyle`, pass through `channels` / `edits` / `constraints`, and set `discreteScale` when discrete data needs a band or a point. Don’t set mark-wide `pointerEvents` — leave that to the engine. To let a datum’s [derived channels (`{ fn }`)](/concepts#derived) see their row index and the dataset, pass the optional `index` and `data` args when you resolve per datum: `resolveStyle(scales, channels, d, defaults, i, data)`.

<Section id="edit-ladder" title="The edit ladder">

Reach for the smallest tool that fits:

1. **Stock edit** — `move`, `resize`, `create`, … when the behaviour already exists.
2. **`custom(fn)`** — one-off `apply`, same gesture/pick defaults.
3. **`makeEdit(spec)`** — full descriptor control (`type`, `pick`, `cardinality`, `name`, …).
4. **`registerDriver`** — multi-event lifecycle / a new `pick` value.

`apply` is pure: return a new datum, a new array, or `undefined`. Prefer `invertChannel` over hand-rolled `scale.invert`.

<Section id="apply-array" title="When apply returns a full array">

Most edits return one datum and the engine splices it at `index`. Whole-dataset edits return a new array — reorder, create, remove. `rank()` is the built-in example: drag swaps discrete rank slots.

<Section id="drivers" title="Register a driver">

Stock `pick` values (`direct`, `nearest`, `plane`, `probe`, `sweep`, …) already cover most lifecycles. When you need a new one, register a driver: `wants(edit)` claims by `pick`, `onEvent` owns hover/click/drag, and `runEdit` / `previewEdit` commit or preview through the same constraint and undo path. Set `selects: true` if `guide: true` should draw a snap ring.

<Section id="glyphs" title="Glyphs: prefer a group of marks">

If a glyph’s handles map to distinct **fields** of a row, build it as a [`composite`](/marks/composite) — ordinary marks over the same rows — so direct-pick dispatch keeps one handle from touching another. Only when several handles must live on **one** feature over **one** datum (positions are derived, not fields) do you need `node.channel` plus a `when` guard. Reach for that last.

A whole-dataset plane gesture (`create`, `rotate`, …) belongs on **exactly one** mark — two marks with the same plane edit would fire twice per click.

</Section>

<Section id="helpers" title="Helpers to reuse">

Don’t reinvent proximity or minting:

- `invertChannel` / `recenterSpan` / `markCenter` — pointer ↔ data for positional edits
- `schemaDefaults(schema)` — mint a row with every field set to its default (or `null`)
- `edit.nextSeriesKey(data, seriesField)` — next free series id for a new line
- `nearestMark` / `nearestSeries` / `nearestMarkOnAxis` — shared distance pickers

For create-like customs, set `cardinality: "append"` so constraints know which row is new. For external UI, give the edit a `name` and drive it with [`el.control(name)`](/editing/external-controls).

</Section>
