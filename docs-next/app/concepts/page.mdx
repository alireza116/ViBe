<Section id="one-dataset" title="Marks are views over a data schema, that can be encoded and edited.">

A chart elicits exactly one dataset from a user who interacts with and edits the chart. Each mark is a _view_ over a dataset: it encodes some features to specific channels and, where a channel carries an `edit`, writes them back.

  code={twoMarksOneDataset}
  title="Two marks, one dataset"
  blurb="Bars and points share the dataset and schema. The point mark carries the drag edit and writes the 'n' field over the y channel, the bar mark updates when the data commits."
  try={
    <>
      <b>Drag</b> a point — both marks update from the same rows.
    </>
  }
/>

</Section>

<Section id="schema" title="Schema owns type and domain">

Since elicitation, usually starts with an empty chart, the dataset is empty and there is nothing to visualize. So we need a data schema. The `schema` defines What each field _is_ — and its **domain**. You define it once. Marks define can define the scale through which they are encoded, and if no scale is defined, the scale is inferred. Details on [Data schema](/schema) and [Scales & channels](/scales).

<Example
  code={schemaOwnsTheDomain}
  title="Schema owns the domain"
  blurb="Nothing names a scale. Schema says cat is categorical and n is quantitative; the bar needs an interval for discrete data — band and linear fall out."
  try={
    <>
      <b>Drag</b> a bar — the edit inverts through the derived scale.
    </>
  }
/>

</Section>

<Section id="channels" title="Channel forms">

Marks encode data through specific channels through a scale. a visual-space constant (`{ value }`), a data-space constant that still goes through the scale (`{ datum }`), or a raw unscaled field (`{ field, scale: null }`). These four stay serializable. The same surface (`x`, `y`, `fill`, `stroke`, `size`, …) is available on every mark, plus top-level style shorthands.

<Example
  code={fieldDrivenColourOrdinalPalette}
  title="Field-driven colour (ordinal palette)"
  blurb={
    'channels: { fill: { field: "kind" } } — a category drives fill through the ordinal palette.'
  }
/>

<Example
  code={constantStyleShorthands}
  title="Constant style shorthands"
  blurb="stroke, strokeWidth, opacity as top-level shorthands on a point — a constant channel needs no scale."

<Section id="derived" title="Derived channels (fn)">

Instead of using scales to derrive visual values, sometimes it's just easier to derrive visual values from functions. a conditional, a lookup across several fields — a fifth form computes the channel per datum: `fill: { fn: d => d.x > 50 ? "red" : "blue" }` (the top-level shorthand `fill: d => …` desugars to it). The function runs in **visual space** — its return value is used as-is (on `y` it is a pixel), never through a scale.

A derived channel is **read-only**: it is opaque to the edit layer, so it can't carry an `edit`. Put the edit on the source field (here `x`) — because every mark re-derives from the committed rows on each render, the fill recomputes for free, live during the drag. On a line or area, a paint fn (`fill`/`stroke`) runs once per **series**, against that series' first row.

<Example
  code={thresholdFillFn}
  title="Threshold fill (derived)"
  blurb={
    'fill: { fn: d => d.x > 50 ? "#dc2626" : "#2563eb" } — the drag writes x; the colour re-derives on every render.'
  }
  try={
    <>
      <b>Drag</b> a point across the dashed line — its colour flips.
    </>
  }
/>

</Section>

<Section id="edits" title="Where edits live">

Encoding maps data → visual; an edit on a channel maps a gesture → data, back through the same scale. Place it on the channel (`y: { field, edit: drag() }`) for a single-channel write, or at mark level (`edits: [...]`) for a joint gesture. Full catalogue on [Editing](/editing).

<Example
  code={markLevelDrag}
  title="Mark-level drag (x and y together)"
  blurb={
    'edits: [drag({ channels: ["x", "y"] })] — one gesture writes both fields.'
  }
  try={
    <>
      <b>Drag</b> a point in two dimensions.
    </>
  }
/>

</Section>

<Section id="constraints" title="Constraints are dataset invariants">

A constraint is a pure rule over **the dataset**. It holds no matter which edit fired it, or which mark declared it — mark-level `constraints` is sugar the engine promotes. It can reject a proposal or repair it (return corrected rows). Catalogue on [Constraints](/constraints).

<Example
  code={clampedDrag}
  title="Clamped drag"
  blurb="clamp keeps n in [10, 50]. guide: true draws the live bound while you drag."
  try={
    <>
      <b>Drag</b> a bar — it stops at 10 and 50.
    </>
  }
/>

</Section>
