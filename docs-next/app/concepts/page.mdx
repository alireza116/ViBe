# Core concepts

<Section id="one-dataset" title="Marks are views over one dataset">

A chart elicits one dataset. Each mark is a _view_ over that dataset: it encodes some fields onto visual channels, and where a channel carries an `edit`, it writes them back. Two marks over the same rows stay in sync — an edit on one commits to the shared data, and the other re-renders from it.

  blurb="Bars and points share the dataset and schema. The point mark carries the drag edit and writes the 'n' field over the y channel, the bar mark updates when the data commits."
  try={
    <>
      <b>Drag</b> a point — both marks update from the same rows.
    </>
  }
/>

</Section>

<Section id="schema" title="The schema owns type and domain">

Elicitation often starts from an empty chart, so the data can't tell you what a field is or how wide its axis runs. The `schema` states this once: each field's **type** (`quantitative`, `categorical`, `ordinal`, or `temporal`) and its **domain**. A mark may name the scale a field encodes through; when it doesn't, the scale is inferred from the type. See [Data schema](/schema) and [Scales & channels](/scales).

  title="Schema owns the domain"
  blurb="Nothing names a scale. Schema says cat is categorical and n is quantitative; the bar needs an interval for discrete data — band and linear fall out."
  try={

      <b>Drag</b> a bar — the edit inverts through the derived scale.

/>

</Section>

<Section id="channels" title="Channels">

A mark encodes data onto channels — `x`, `y`, `fill`, `stroke`, `size`, and so on. Every mark exposes the same channel set, plus top-level style shorthands. A channel takes one of four serializable forms:

- `{ field: "n" }` — a field, mapped through its scale.
- `{ value: 25 }` — a visual-space constant; on `y` it is pixel 25, skipping the scale.
- `{ datum: 25 }` — a data-space constant, mapped through the scale; on `y` it lands where the value 25 sits.
- `{ field: "n", scale: null }` — a raw field, used as-is with no scale.

<Section id="derived" title="Derived channels">

A fifth form computes the channel from a function, one datum at a time: `fill: { fn: d => d.x > 50 ? "red" : "blue" }` (the shorthand `fill: d => …` desugars to it). The function runs in **visual space** — its return value is used as-is (on `y`, a pixel), never through a scale.

A derived channel is **read-only**. It is opaque to the edit layer, so it can't carry an `edit`; put the edit on the source field instead (here `x`). Every mark re-derives from the committed rows on each render, so the colour recomputes live during the drag. On a line or area, a paint function (`fill`/`stroke`) runs once per **series**, against that series' first row.

<Section id="edits" title="Edits">

An edit is the inverse of an encoding. Encoding maps data → visual; an edit maps a gesture → data, back through the same scale. Put it on a channel (`y: { field, edit: drag() }`) to write one field, or at mark level (`edits: [...]`) for a gesture that writes several at once. Full catalogue on [Editing](/editing).

</Section>

<Section id="constraints" title="Constraints">

A constraint is a pure rule over **the dataset**. It holds no matter which edit fired it or which mark declared it — a mark's `constraints` is sugar the engine promotes to the whole dataset. A constraint can reject a proposed edit or repair it by returning corrected rows. Catalogue on [Constraints](/constraints).
